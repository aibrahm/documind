- **`src/lib/tools/extract-key-terms.ts`** (~210 lines):

  - Includes a hint in the response telling the model to call `financial_model` separately if it wants NPV/IRR comparisons (tool chaining nudge)

## Decisions Made

1. **GPT-4o-mini, not Claude, for the extraction LLM call** — Same pattern as the existing librarian/router/memory code. It's cheap, structured-JSON output is supported via `response_format`, and the extraction task is mechanical enough that a small model handles it well. Saves cost vs calling Claude recursively from inside a Claude tool.
2. **Additive merge with new-wins-on-conflict** — The spread `{ ...previous, ...extracted }` means existing fields in the negotiation are preserved IF the fresh extraction omits them, but a fresh extraction overwrites a stale value when both are present. This is the right default for "re-extract from updated documents" — the latest read should be the authoritative one.
3. **Free-form key_terms, no validation** — The negotiation `key_terms` JSONB is intentionally schema-free per the Phase 03 design. The extraction prompt nudges the model toward common fields but accepts whatever the documents actually contain.
4. **30 KB context budget** — Truncates per-document at the chunk boundary rather than truncating the final string mid-token. Predictable and safe for GPT-4o-mini's context window.
5. **`compare_deals` doesn't compute NPV/IRR** — The note in the response tells the model to call `financial_model` separately for each negotiation if the user wants those metrics. Keeps the comparison tool single-purpose and lets the model chain tools naturally.
6. **Max 5 negotiations per comparison** — Matrices wider than 5 columns become unreadable in the UI and overwhelm the model's output budget. Hard limit, returns an error above 5.
7. **Bundled commit for both tools** — Splitting would have committed broken intermediate states (the dispatch loop references both handlers).
