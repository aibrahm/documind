<objective>
Centralize shared TypeScript types and normalize inconsistent naming across the chat surface.

Today the `Source` type is exported from `src/app/page.tsx` (a page component, wrong layer) and re-imported by `chat-message.tsx`. `AttachmentMeta` is also defined there. `PinnedItem` lives in `chat-input.tsx` and is re-imported by `page.tsx` and `chat-message.tsx`. This creates circular-ish import patterns and makes the page component the canonical home of types it shouldn't own.

Naming-wise: `pendingFile` vs `pending` vs `attachment` are used interchangeably in `chat-input.tsx`. Source variables are called `docSources`, `pinnedDocSources`, `webSources`, `additionalWebSources`, `allSources`, `pinnedDocPills` in the same function — confusing.

Purpose: Page-level files import shared types from `src/lib/types.ts`, not from each other. Naming is consistent so future contributors don't have to guess.

Output: New `src/lib/types.ts` exporting `Source`, `AttachmentMeta`, `PinnedItem`. All consumers updated. State variables renamed for clarity.
</objective>

// src/lib/types.ts

// ── Source ──
// A retrieved document chunk OR a web result OR a pinned document pill.
// Discriminated union by `type` field.
export type Source =
  | {
      id: string;
      type: "document";
      title: string;
      pageNumber: number;
      sectionTitle?: string | null;
      documentId: string;
    }
  | {
      id: string;
      type: "web";
      title: string;
      url: string;
    };

// ── AttachmentMeta ──
// Metadata about an ephemeral chat attachment, persisted on the user message
// so the chip renders correctly when reloading conversation history.
export interface AttachmentMeta {
  title: string;
  pageCount: number;
  size: number;
}

// ── PinnedItem ──
// User-pinned reference from the @ picker. Either a specific document
// (full chunks loaded as primary evidence) or an entity (resolved to
// document mentions via name search).
export interface PinnedItem {
  kind: "document" | "entity";
  id: string;
  label: string;
  type?: string;        // entity type (company/person/place/...) or document type (memo/contract/...)
  doc_count?: number;   // entities only
}

Two coordinated renames. Both are mechanical, just be thorough.

**A. `pendingFile` → `pendingAttachment` in `chat-input.tsx`.**

Currently the file uses `PendingFile` interface, `pending` state, `pending` parameter names, and various combinations. Standardize on `PendingAttachment`:

1. Rename the interface: `interface PendingFile` → `interface PendingAttachment`
2. Rename the state: `const [pending, setPending] = useState<PendingFile[]>([])` → `const [pendingAttachments, setPendingAttachments] = useState<PendingAttachment[]>([])`
3. Update every usage: `pending.map(...)`, `pending.filter(...)`, `setPending(...)` → use the new names
4. The user-facing concept is "attachment" (we call them attachments in the message metadata), so the in-progress version is "pending attachment". The state is `pendingAttachments` (plural). The interface is `PendingAttachment` (singular).

This is a pure mechanical rename within ONE file. Use `replace_all: true` for `setPending(` → `setPendingAttachments(`. Same for the others. Verify no references leak by grep'ing the file after.

**B. Source variable naming in `src/app/api/chat/route.ts` and `src/app/api/chat/[id]/route.ts`.**

The current naming inside the `try` blocks is:
- `docSources` — chunks from hybrid search (document evidence)
- `pinnedDocSources` — chunks from pinned documents (primary evidence)
- `webSources` — web search results
- `additionalWebSources` — web sources discovered via Claude tool use (should merge into webSources)
- `allSources` — concatenation
- `pinnedDocPills` — UI projection of pinned chunks (one per page)

Rename for clarity (apply in BOTH files):
- `docSources` → `documentEvidence`
- `pinnedDocSources` → `pinnedEvidence`
- `webSources` → `webEvidence` (the existing list)
- `additionalWebSources` → KEEP this name (it's a separate temporary list inside the Claude tool-use closure that gets MERGED into `webEvidence` later)
- `allSources` → `evidenceSources`
- `pinnedDocPills` → `pinnedSourcePills`
- `seenPinnedPages` → `seenPinnedPages` (keep — local var, fine)
- `seenPinnedDocs` → was already removed when we switched to per-page dedup

Mechanical steps per file:
1. Use `replace_all` for each old → new name
2. Run `npx tsc --noEmit` after each file to catch typos
3. Verify the renamed variables still match the existing API responses (no field renames, just variable names)

What to avoid:
- Do NOT rename SSE event types (`type: "sources"`, `type: "text"`, `type: "tool"`) — those are part of the streaming protocol the UI consumes
- Do NOT rename anything in the SSE payload field names — UI-facing wire format stays stable
- Do NOT rename types in `database.types.ts`
- Do NOT rename `claude-with-tools.ts` internals — that file is stable
