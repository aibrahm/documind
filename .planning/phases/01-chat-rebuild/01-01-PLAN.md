<objective>
Rebuild DocuMind as a multi-turn chat experience with fluid intelligence routing.

Purpose: Transform from single query→response into a conversational interface where follow-ups work, context carries forward, and the system knows WHEN to apply heavy doctrine analysis vs just answering naturally.

Output: New database schema for chat sessions with messages, new chat API with message history support, intelligence router that picks the right mode automatically.
</objective>

<context>
Current state:
- Single query → single response model
- Doctrine system always applies full scoring (too rigid)
- No message history sent to LLM
- Conversations table stores single query+response pairs

Target state:
- Multi-turn chat like Claude/ChatGPT
- Each session is a thread of messages (user + assistant alternating)
- System automatically decides: simple answer, search, or full doctrine analysis
- Follow-up questions use previous context
- Conversations persist with all messages

Key insight: The doctrine scoring should ONLY activate when the user is asking for evaluation/analysis. Simple questions ("what's in this document?", "summarize page 5", "translate this") should get fluid, natural answers with document context but WITHOUT the rigid scoring format.
</context>

```sql
-- Messages within conversations
CREATE TABLE messages (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  conversation_id UUID NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
  role TEXT NOT NULL CHECK (role IN ('user', 'assistant', 'system')),
  content TEXT NOT NULL,
  metadata JSONB DEFAULT '{}',
  -- metadata can contain: sources, classification, model, doctrines, attachments, search_results
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_messages_conversation ON messages (conversation_id, created_at);

ALTER TABLE messages ENABLE ROW LEVEL SECURITY;
CREATE POLICY "read_messages" ON messages FOR SELECT TO authenticated USING (true);
CREATE POLICY "write_messages" ON messages FOR INSERT TO authenticated WITH CHECK (true);
```

<task type="auto">
  <name>Task 2: Intelligence router — decide response mode automatically</name>
  <files>src/lib/intelligence-router.ts</files>
  <action>
Create a new module that analyzes the user's message and conversation history to decide HOW to respond. This replaces the manual "search" vs "analyze" toggle.

Three response modes:
1. **casual** — Simple question, follow-up, clarification, translation, summary request. Use GPT-4o-mini with document context but NO doctrine. Fast, conversational.
2. **search** — User wants to find specific information. Run hybrid search, return results with citations. No analysis.
3. **deep** — User wants evaluation, risk analysis, comparison, scoring. Activate full doctrine pipeline with Claude Sonnet.

The router should consider:
- The current message text
- Previous messages in the conversation (to detect follow-ups)
- Keywords and intent signals
- Whether the user explicitly asked for analysis/evaluation

```typescript
export type ResponseMode = "casual" | "search" | "deep";

export interface RoutingDecision {
  mode: ResponseMode;
  shouldSearch: boolean;        // whether to retrieve documents
  doctrines: string[];          // which doctrines to activate (empty for casual)
  searchQuery: string;          // optimized search query (may differ from user message)
  reasoning: string;            // why this mode was chosen (for debugging)
}
```

Use GPT-4o-mini for routing (single fast call). The prompt should be:
- Given the conversation history and latest message, classify the intent
- If it's a follow-up, the search query should incorporate previous context
- If the user says "analyze", "evaluate", "assess", "score", "compare" → deep mode
- If the user says "find", "search", "where", "show me" → search mode
- Everything else → casual mode (but still retrieve relevant docs if the question is about documents)

Do NOT make this a simple keyword match. Use the LLM to understand intent in context.

  <action>
Create new chat API routes that replace the current query/search routes:

**POST /api/chat** — Create a new chat session
- Body: { message: string, attachments?: [] }
- Creates conversation + first user message
- Routes through intelligence router
- Streams response via SSE
- Saves assistant message when complete
- Returns SSE stream with events:
  - {type: "session", id: conversationId} — first event
  - {type: "routing", mode, doctrines, searchQuery} — routing decision
  - {type: "sources", sources: [...]} — retrieved documents (if search was done)
  - {type: "text", content: "delta"} — streaming response text
  - {type: "done"} — complete

**POST /api/chat/[id]** — Continue an existing chat
- Body: { message: string, attachments?: [] }
- Loads conversation history (last N messages)
- Routes through intelligence router WITH history context
- Same SSE stream format
- Saves both user message and assistant response

**GET /api/chat/[id]/messages** — Load all messages for a conversation
- Returns { messages: [{id, role, content, metadata, created_at}] }

Key implementation details:
- Send last 10 messages as conversation history to the LLM
- For casual mode: use GPT-4o-mini with retrieved doc chunks as context, NO doctrine
- For search mode: run hybridSearch, format results, respond with GPT-4o-mini
- For deep mode: full doctrine pipeline with Claude Sonnet (existing logic from query route)
- Always save messages to DB after completion
- The system prompt should adapt based on mode:
  - Casual: "You are DocuMind, a knowledgeable document assistant. Answer naturally based on the documents. Cite sources as [DOC-N] when referencing specific content."
  - Search: (no LLM needed, just return search results)
  - Deep: Full doctrine system prompt (existing)
