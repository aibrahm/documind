<objective>
Extract the shared chat-turn logic from `/api/chat/route.ts` and `/api/chat/[id]/route.ts` into a single `src/lib/chat-turn.ts` helper. The two routes are currently ~80% duplicate, which means every change has to be made twice and the two paths drift over time.

After the refactor, both routes become thin wrappers (~50 lines each) that:
1. Parse and validate the request body
2. Resolve the conversation (create new vs load existing)
3. Call `runChatTurn(...)` with the unified arguments
4. Return the streaming response

All the rest — entity detection, pinned doc/entity resolution, evidence package construction, system prompt building, Claude tool-use loop with GPT fallback, message persistence, fire-and-forget memory extraction — lives in `runChatTurn`.

Purpose: Eliminate the dual-source-of-truth problem. Phase 03+ will add project-scoped chat behavior; that change should land in ONE place, not two.

Output: New `src/lib/chat-turn.ts` (~350 lines) + dramatically slimmed routes. Behavior identical to before — the entire flow is verified end-to-end with Playwright after the refactor.
</objective>

```typescript
// src/lib/chat-turn.ts

import type { SupabaseClient } from "@supabase/supabase-js";
import type { Database } from "@/lib/database.types";
import type { Source } from "@/lib/types";
// ... (other imports)

export interface InboundAttachment {
  title: string;
  content: string;
  pageCount?: number;
  size?: number;
}

export interface RunChatTurnArgs {
  conversationId: string;       // already-resolved conversation id (caller creates or loads)
  userMessage: string;
  attachments: InboundAttachment[];
  pinnedDocumentIds: string[];
  pinnedEntityIds: string[];
  history: Array<{ role: "user" | "assistant" | "system"; content: string }>;
  // SSE controller hooks (so the route stays in charge of the response stream)
  emit: (eventType: string, payload: Record<string, unknown>) => void;
}

export interface RunChatTurnResult {
  fullText: string;
  routing: { mode: string; doctrines: string[] };
  evidenceSources: Source[];   // for the saved message metadata
  modelUsed: string;
}

export async function runChatTurn(args: RunChatTurnArgs): Promise<RunChatTurnResult> {
  // 1. Routing (router + parallel memory retrieval)
  // 2. Entity detection in user message
  // 3. Pinned reference resolution (docs + entities → chunks)
  //    - includes the name-based search fallback for entities
  //    - includes the dual-header logic (PINNED ENTITY vs PINNED DOCUMENTS)
  // 4. Hybrid search (with entity scoping if no pins)
  // 5. Optional web search if router says shouldWebSearch
  // 6. Build evidence package (pinned first, then docs, then web, then attachments)
  // 7. Build LLM messages array (system + history + user-with-evidence)
  // 8. Build system prompt (deep mode → doctrine, casual mode → casual prompt)
  // 9. Emit `routing` and `sources` SSE events via the emit callback
  // 10. Stream the LLM response:
  //     - Deep mode: runClaudeWithTools (autonomous tool use)
  //     - Casual mode: gpt-4o-mini direct stream
  //     - Either way, emit `text` events for each delta
  //     - For deep mode also emit `tool` events from the onToolStart/onToolEnd callbacks
  //     - For deep mode, append additionalWebSources to the saved evidenceSources
  // 11. Emit `done`
  // 12. Persist the assistant message to the messages table
  // 13. Fire-and-forget memory extraction (don't await)
  // 14. Return { fullText, routing, evidenceSources, modelUsed }
}
```

How to actually port the logic:

1. Open `src/app/api/chat/route.ts` and read the entire `start(controller)` body — that's the source of truth.
2. Copy it into `runChatTurn`, replacing every `controller.enqueue(encoder.encode(\`data: ${JSON.stringify(...)}\\n\\n\`))` with `args.emit(eventType, payload)`. The `emit` callback handles serialization on the route side.
3. Replace `convo.id` with `args.conversationId`.
4. Replace `userMessage` with `args.userMessage`.
5. Replace `attachments` with `args.attachments`.
6. Replace `pinnedDocumentIds` with `args.pinnedDocumentIds` (same for entities).
7. Replace the local `messages: Array<...>` (history) with `args.history` for building llmMessages — currently the new route has empty history, the [id] route loads from DB. After refactor: the route resolves history before calling runChatTurn and passes it in.
8. Persist the assistant message at the end (insert into `messages` table with full metadata).
9. Fire-and-forget memory extraction at the end (`extractMemories(...).then(storeMemories).catch(...)`).
10. Return the result tuple.

The `emit` callback signature is:
```typescript
emit: (type: string, payload: Record<string, unknown>) => void
```
Inside `runChatTurn`, call it like:
```typescript
args.emit("routing", { mode: routing.mode, doctrines: routing.doctrines, reasoning: routing.reasoning });
args.emit("sources", { sources: [...evidenceSources, ...] });
args.emit("text", { content: delta });
args.emit("tool", { status: "start", name: "web_search", query });
args.emit("done", {});
```

The route side wraps `emit` like:
```typescript
const emit = (type: string, payload: Record<string, unknown>) => {
  controller.enqueue(
    encoder.encode(`data: ${JSON.stringify({ type, ...payload })}\n\n`),
  );
};
```

- Do NOT skip the memory extraction at the end of either route.
- Do NOT touch `claude-with-tools.ts`.

```typescript
import { NextRequest, NextResponse } from "next/server";
import { supabaseAdmin } from "@/lib/supabase";
import { runChatTurn, type InboundAttachment } from "@/lib/chat-turn";
import { logAudit } from "@/lib/audit";

export const maxDuration = 60;

export async function POST(request: NextRequest) {
  // 1. Parse + validate body (existing validation logic, unchanged)
  let body: {
    message?: string;
    attachments?: InboundAttachment[];
    pinnedDocumentIds?: string[];
    pinnedEntityIds?: string[];
  };
  try { body = await request.json(); } catch {
    return NextResponse.json({ error: "Invalid JSON" }, { status: 400 });
  }
  const { message, attachments: rawAttachments, pinnedDocumentIds: rawPinDocs, pinnedEntityIds: rawPinEnts } = body;
  // ... existing validation, attachment filter, pin filter ...
  
  // 2. Create conversation row
  const title = userMessage.length > 60 ? userMessage.slice(0, 60) + "…" : userMessage || "(attachment)";
  const { data: convo, error: convoErr } = await supabaseAdmin
    .from("conversations")
    .insert({ title, mode: "chat", query: userMessage })
    .select("id")
    .single();
  if (convoErr || !convo) return NextResponse.json({ error: "Failed to create session" }, { status: 500 });
  
  // 3. Save the user message immediately (so the conversation is queryable even if streaming fails)
  const attachmentMeta = attachments.length > 0
    ? attachments.map((a) => ({ title: a.title, pageCount: a.pageCount ?? 0, size: a.size ?? 0 }))
    : undefined;
  await supabaseAdmin.from("messages").insert({
    conversation_id: convo.id,
    role: "user",
    content: userMessage,
    metadata: attachmentMeta ? { attachments: attachmentMeta } : {},
  });
  
  // 4. Stream response via runChatTurn
  const encoder = new TextEncoder();
  const stream = new ReadableStream({
    async start(controller) {
      const emit = (type: string, payload: Record<string, unknown>) => {
        controller.enqueue(encoder.encode(`data: ${JSON.stringify({ type, ...payload })}\n\n`));
      };
      
      // Always send session id first (UI uses it to switch from /api/chat to /api/chat/[id])
      emit("session", { id: convo.id });
      
      try {
        await runChatTurn({
          conversationId: convo.id,
          userMessage,
          attachments,
          pinnedDocumentIds,
          pinnedEntityIds,
          history: [],   // new conversation, no history
          emit,
        });
      } catch (err) {
        console.error("runChatTurn failed:", err);
        emit("error", { message: "Failed to generate response" });
      }
      
      logAudit("query", { conversationId: convo.id }).catch(console.error);
      controller.close();
    },
  });
  
  return new Response(stream, {
    headers: { "Content-Type": "text/event-stream", "Cache-Control": "no-cache", Connection: "keep-alive" },
  });
}
```
