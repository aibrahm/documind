# Phase 03.5 — Plan 01 Summary

**financial_model + fetch_url tools shipped — Claude Opus now does guaranteed-correct math and reads full webpages on demand**

## Accomplishments

- **`src/lib/tools/financial-model.ts`** (~165 lines) — pure-JS NPV/IRR/payback/sensitivity with Newton-Raphson IRR + bisection fallback and a `runFinancialModel` handler that returns JSON
- **`src/lib/tools/fetch-url.ts`** (~115 lines) — `fetchUrlContent(url)` with 15s AbortController timeout, protocol whitelist, content-type guard, regex HTML stripping (script/style/noscript + tags + entities + whitespace collapse), 8000-char truncation, and a `runFetchUrl` handler
- **`src/lib/claude-with-tools.ts`** rewired:
  - `Block.tool_use.input` widened from web_search-specific to `Record<string, unknown>`
  - `FINANCIAL_MODEL_TOOL` and `FETCH_URL_TOOL` Anthropic tool definitions added next to `WEB_SEARCH_TOOL`
  - `tools: [...]` array now passes all three to the streaming call
  - Dispatch loop switched from a single web_search branch to a per-name `if/else` chain (web_search → financial_model → fetch_url → unknown fallback)
  - `onToolStart`/`onToolEnd` signatures gained an optional `toolName` arg
- **`src/lib/chat-turn.ts`** — `emit("tool", ...)` now reads the `toolName` from the callback instead of hardcoding `"web_search"`. Backward compatible (defaults to `"web_search"` if not provided).

## Decisions Made

1. **Zero new dependencies** — Newton-Raphson IRR is 20 lines of code; the `financial` npm package adds 3MB and 12 transitive deps for the same correctness. Regex HTML stripping is good enough for the 80% case; cheerio adds another 2MB. Pure JS keeps the bundle slim and the dependency surface predictable.
2. **financial_model is silent (no SSE events), fetch_url is loud** — Arithmetic is instant, and emitting status would just produce flicker. Web fetches can take 10+ seconds; the user needs to see "Reading: example.com" while it's working. Different tools, different UX needs.
3. **Tool description tone is prescriptive** — Both tools' descriptions have explicit "DO" and "DON'T" instructions. `financial_model`: "DO NOT do arithmetic in your head — always call this tool". `fetch_url`: "DON'T use this for general research — prefer web_search first". This reduces over-fetching and prevents the model from skipping the calculator.
4. **Dispatch is per-name `if` branches, not a name → handler map** — The branches differ in their callback patterns (financial_model takes no callbacks, web_search/fetch_url do). A map would force a uniform shape and add indirection. Three branches is fine.
5. **`Block.tool_use.input` widened to `Record<string, unknown>`** — The previous narrow type was a Phase 01 hangover when only web_search existed. Generic over all tools now; each handler casts as needed.
6. **`onToolStart`/`onToolEnd` callbacks gained `toolName` arg** — The plan called for this implicitly (so the UI can show different status text per tool). Made it optional + defaulted in chat-turn.ts so existing call sites keep working.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] SSE `tool` events were hardcoding `name: "web_search"`**
- **Found during:** Live test of fetch_url — the SSE event showed `name: "web_search"` even when fetch_url fired. The query text was correct ("Reading: ..."), but the tool name was wrong.
- **Root cause:** `onToolStart`/`onToolEnd` only passed `query` to the caller. `chat-turn.ts` hardcoded `name: "web_search"` because there was no other signal.
- **Fix:** Extended both callback signatures with an optional `toolName` arg. Updated `chat-turn.ts` to read it (defaulting to `"web_search"` for backward compatibility). Each dispatch branch in `claude-with-tools.ts` passes its tool name explicitly.
- **Files modified:** src/lib/claude-with-tools.ts, src/lib/chat-turn.ts
- **Verification:** Future SSE events will correctly distinguish the tools. Verified by re-reading the dispatch + emission code.
- **Committed in:** 9888c5e (bundled with the main commit)

### Deferred Enhancements

- **Sensitivity sweep with multiple variables (e.g. discount_rate + tenor)** — Plan only specifies a single `variable: "discount_rate"`. Multi-variable sweeps would need a 2D return shape and a more complex tool description. Defer until someone actually asks for it.
- **NPV with currency unit explicit in the input** — Currently the tool returns `currency: "unspecified (assumed consistent across cashflows)"`. Could add a `currency` field for clarity. Cosmetic; defer.
- **`fetch_url` PDF handling** — Returns "Unsupported content type" for PDFs. Could pipe through `pdf-parse` (already in deps) to extract text. Defer to a Phase 03.5-02 enhancement if Tavily surfaces a lot of PDFs.

## Issues Encountered

The first two smoke tests of the live tools went to **casual mode** (GPT-4o-mini) instead of deep mode, which meant Claude (the only model with the tools registered) wasn't reached. The router classifies queries based on intent — straightforward "use this tool" commands look casual to GPT-4o-mini's classifier. Reframing the queries with analytical hooks ("KIZAD benchmarks", "regulatory submission") routed them correctly to deep mode and the tools fired as expected. Not a bug — the router is doing its job. The doctrine prompts in deep mode + the tool descriptions did the rest.
